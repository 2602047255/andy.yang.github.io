回调地狱的缺点：

  • 代码耦合性太强，牵一发而动全身，难以维护
  • 大量的冗余代码相互嵌套，代码的可读性变差

Promise 基本概念

  • Promise 是一个构造函数
    • 我们可以创建 Promise 的实例 const p = new Promise()
    • new 出来的 Promise 实例对象，代表一个异步操作
  • Promise.prototype 上包含一个 .then 方法
    • 每一次 new Promise() 构造函数得到的实例对象，都可以通过原型链的方式访问到 .then() 方法
    • 例如：p.then()
  • .then() 方法用来预先指定成功和失败的回调函数
    • p.then(成功的回调函数, 失败的回调函数)
    • p.then(result => {}, error => {})
    • 调用 .then() 方法时，成功的回调函数是必选的，失败的回调函数是可选的
  • .then() 方法的特性
    • 如果上一个 .then() 方法返回了一个新的 Promise，则可以通过下一个 .then() 继续进行处理
    • 通过 .then() 方法的链式调用，就可以解决回调地狱的问题
  • Promise.prototype 上包含一个 .catch 方法
    • 在 Promise 的链式操作中，如果发生错误，可以使用 .catch() 方法进行捕获和处理
    • 只要发生了错误，下面的 .then() 方法将不再继续执行，就会直接进入 .catch() 方法
    • 如果不希望前面的错误导致后续的 .then 无法正常执行，可以将 .catch 的调用往前提

Promise.all

Promise.all() 方法会发起并行的 Promise 异步操作，等所有的异步操作全部结束后才会执行 .then 操作（等待机制）

Promise.race

Promise.race() 方法会发起并行的 Promise 异步操作，只要任何一个异步操作完成，就会立即执行 .then 操作（赛跑机制）

基于 Promise 的函数封装

  • 成功的结果可以放在 resolve 里面，这样可以被 .then 方法的成功的回调函数捕获
  • 失败的结果可以放在 reject 里面，这样可以被 .then 方法的失败的回调函数或者被 .catch 捕获

单线程

  • JavaScript 是一门单线程执行的编程语言，简单来说就是，同一时间只能做一件事情
  • JavaScript 执行线程 — 任务1(执行中) — 任务2(待执行) — 任务3(待执行) — 任务N(待执行)
  • 如果前一个任务非常耗时，后续的任务就不得不一直等待，从而导致程序假死的问题

同步任务和异步任务

为了防止某个耗时任务导致程序假死的问题，JavaScript 把待执行任务分为两类：

  • 同步任务
    • 又叫非耗时任务，指的是在主线程上排队执行的那些任务
    • 只有前一个任务执行完毕，才能执行下一个任务
  • 异步任务
    • 又叫耗时任务，异步任务由 JavaScript 委托给宿主环境进行执行
    • 当异步任务执行完成后，会通知 JavaScript 主线程执行异步任务的回调函数

同步任务和异步任务的执行过程

  1. 同步任务由 JavaScript 主线程次序执行
  2. 异步任务委托给宿主环境（浏览器、Node）执行
  3. 已完成的异步任务对应的回调函数，会被放到任务队列中等待执行
  4. JavaScript 主线程的同步任务全部执行完后，会读取任务队列中的回调函数，次序执行
  5. JavaScript 主线程不断重复上面的第4步

宏任务与微任务

JavaScript 把异步任务又做了进一步的划分，异步任务又分为两类，分别是：

  • 宏任务：
    • 异步的 ajax 请求
    • setTimeout、setInterval
    • 文件操作
    • 其它宏任务
  • 微任务：
    • Promise 中的 .then、.catch、.finally
    • process.nextTick
    • Object.observe
    • 其它微任务

宏任务与微任务的执行顺序

假设先执行宏任务，当第一个宏任务执行结束后，会先判断是否有微任务，每一个宏任务执行完之后，都会检查是否存在待执行的微任务

如果有，就执行所有的微任务，当所有的微任务结束后，再执行下一个宏任务
如果没有，就直接执行下一个宏任务

去银行办业务的场景

  • 张三和李四去银行办业务，首先，需要取号之后进行排队
    • 宏任务队列
  • 假设当前银行网点只有一个柜员有空，张三在办理存款业务时，李四只能等待
    • 单线程，宏任务按次序执行
  • 张三办完存款业务后，柜员询问他是否还想办理其它业务？
    • 当前宏任务执行完毕，检查是否有微任务
  • 张三告诉柜员：想要买理财产品、再办个信用卡、最后再兑换点纪念币
    • 买理财产品 => 第一个微任务
    • 办个信用卡 => 第二个微任务
    • 兑换纪念币 => 第三个微任务
    • 执行微任务，后续宏任务被推迟
  • 张三离开柜台后，柜员开始为李四办理业务
    • 所有微任务执行完毕，开始执行下一个宏任务

经典面试题

为什么正确的输出顺序是：2、4、3、1 呢？这里我们来做一下解析：

  • new Promise() 和 console.log(4) 都属于同步任务，所以会先输出 2、4
  • 检查是否有微任务，.then() 属于微任务，所以输出 3
  • 再执行下一个宏任务，setTimeout() 属于宏任务，最后输出 1
